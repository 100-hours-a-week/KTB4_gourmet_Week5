# 회고 & AI 사용 기록

## 진행한 내용

이번 과제에서는 커뮤니티 서비스의 ERD를 설계하고, 설계한 ERD와 동일한 결과를 만들 수 있도록 DDL을 작성했다.

처음에는 단순히 회원, 게시글, 댓글 정도만 테이블로 나누면 된다고 생각했는데, 실제로 관계를 하나씩 연결해보니 생각보다 고민할 부분이 많았다. 단순히 ERD를 그리는 것이 아니라, 각 데이터가 어떤 테이블에 있어야 하고 테이블끼리 어떤 관계를 가져야 하는지 보는 것이 중요하다는 것을 느꼈다.

---

## 설계 과정

커뮤니티 기능에 필요한 데이터는 크게 회원, 게시글, 댓글, 이미지, 좋아요라고 판단했다. 그래서 최종적으로 다음과 같이 테이블을 나누었다.

* `users`
* `posts`
* `comments`
* `image`
* `post_likes`

가장 고민했던 부분은 좋아요 관계였다. 회원은 여러 게시글에 좋아요를 누를 수 있고, 게시글도 여러 회원에게 좋아요를 받을 수 있기 때문에 처음에는 단순히 회원과 게시글의 N:M 관계라고만 생각했다.

하지만 관계형 데이터베이스에서는 N:M 관계를 직접 표현하기보다 중간 테이블로 풀어야 한다는 것을 다시 확인했다. 그래서 `post_likes` 테이블을 따로 만들고, 회원과 좋아요는 1:N, 게시글과 좋아요도 1:N 관계로 설계했다. 결과적으로 `post_likes`를 통해 회원과 게시글의 N:M 관계를 구현한 구조가 되었다.

이미지 테이블도 설계하면서 수정을 하게 되었다. 처음에는 이미지 테이블을 만들 생각을 못했다. 논리 흐름 관점에서 생각해보니 회원이 사진 한 장만 올리는 것은 말이 안된다 생각을 하여 이미지 테이블을 만들기로 했다. 게시글 하나에 이미지가 여러 개 들어갈 수 있고, 이미지도 생성일, 수정일, 삭제일 같은 정보를 따로 가질 수 있다고 생각하니 게시글 테이블 안에 이미지 정보를 넣는 것은 구조적으로 맞지 않다고 느껴서 수정하게 되었다.

그래서 `image` 테이블을 따로 분리하고, 게시글과 이미지를 1:N 관계로 연결했다. 이 과정에서 단순히 컬럼을 추가하는 것과 테이블을 분리하는 것의 차이를 조금 더 이해하게 되었다.

시간 컬럼의 자료형은 `TIMESTAMP`가 아니라 `DATETIME`을 사용했다. 이번 과제는 로컬 환경에서 커뮤니티 테이블 구조를 설계하는 것이 목적이고, 시간대 변환이나 UTC 기준 저장까지 깊게 다루는 상황은 아니라고 판단했다. 그래서 DB가 시간대를 변환하는 방식보다는 생성일, 수정일, 삭제일 값을 그대로 저장하는 `DATETIME`이 더 단순하고 이해하기 쉽다고 생각했다.

물론 실제 글로벌 서비스처럼 여러 시간대를 고려해야 하는 경우라면 UTC 기준 저장 방식이나 `TIMESTAMP` 사용도 함께 고민해야 할 것 같다. 이번 설계에서는 로컬 개발 환경과 과제 범위를 기준으로 `DATETIME`을 선택했다.

---

## 어려웠던 점

ERDCloud를 사용하는 과정에서도 시행착오가 있었다. 관계선을 연결하면 외래키가 자동으로 생성되는 부분을 처음에는 제대로 기억하지 못해서, FK를 직접 만들어야 하는지 헷갈렸다.

처음에는 수동으로 컬럼을 만들고 관계를 연결하는 과정에서 중복되거나 어색한 부분이 생길 수 있다는 것을 알게 되었다. 이후에는 부모 테이블과 자식 테이블의 관계를 먼저 생각하고, 어떤 컬럼이 FK가 되어야 하는지 확인하면서 다시 정리했다.

또 테이블을 한 번에 완성하지 못했다. 처음 설계한 뒤에도 좋아요 관계, 이미지 관계, 조회수 컬럼 등을 다시 보면서 논리적으로 어색한 부분이 없는지 계속 확인했다. 설계하면서 계속 수정하다 보니, 데이터베이스 설계는 처음부터 한 번에 끝나는 작업이 아니라 관계를 살펴보면서 반복적으로 다듬는 과정이라는 것을 느꼈다.

---

## DDL 작성 과정

DDL 작성에서도 ERDCloud에서 Export한 SQL을 그대로 사용할 수는 없었다. Export된 DDL에는 자료형이나 기본값, 제약조건이 부족한 부분이 있었기 때문에 직접 수정이 필요했다.

기본키에는 `AUTO_INCREMENT`를 적용했고, 생성일에는 `DEFAULT CURRENT_TIMESTAMP`를 사용했다. 이메일과 닉네임에는 중복을 막기 위해 `UNIQUE` 제약조건을 추가했다.

좋아요 테이블에는 같은 회원이 같은 게시글에 중복 좋아요를 누르지 못하도록 `user_id`, `post_id`를 묶은 복합 `UNIQUE` 제약조건을 추가했다. 각각의 컬럼을 따로 `UNIQUE`로 설정하면 회원이 좋아요를 한 번밖에 못 누르거나, 게시글이 좋아요를 하나밖에 못 받는 잘못된 구조가 되기 때문에 두 컬럼의 조합을 기준으로 중복을 막도록 했다.

또 하나 고민했던 부분은 `updated_at`이다. 현재 설계에서는 `ON UPDATE CURRENT_TIMESTAMP`를 사용했기 때문에 게시글의 제목이나 내용 수정뿐만 아니라 조회수 증가처럼 게시글 행이 변경되는 경우에도 `updated_at`이 변경될 수 있다. 이번 설계에서는 게시글 데이터의 마지막 변경 시각으로 보고 그대로 사용했지만, 실제 서비스에서는 제목/내용 수정 시간과 조회수 변경 시간을 분리해서 관리할 수도 있겠다고 생각했다.

---

## 느낀 점

이번 과제를 하면서 테이블 설계가 실제 개발에서 얼마나 중요한지 느꼈다. 테이블 관계가 잘못 잡히면 나중에 API나 서비스 로직을 만들 때 계속 꼬일 수 있겠다는 생각이 들었다.

좋아요와 이미지처럼 처음에는 단순하게 생각했던 부분도 실제 관계를 따져보면 별도 테이블이 필요하거나 제약조건이 필요하다는 것을 알게 되었다. 특히 N:M 관계를 중간 테이블로 분리하는 과정과, DDL에서 제약조건을 통해 데이터 무결성을 보장하는 부분을 중점적으로 학습했다.

이번 과제는 생각보다 수정할 부분이 많았기 때문에 일찍 시작하길 잘했다고 느꼈다. 앞으로도 기능 구현 전에 데이터 구조와 관계를 먼저 꼼꼼하게 보는 습관을 가져야겠다고 생각했다.

# AI 사용 기록

## 사용 목적

이번 과제에서는 커뮤니티 ERD 설계와 DDL 작성 과정에서 헷갈리는 개념을 확인하고, 내가 작성한 설계가 논리적으로 맞는지 검토하기 위해 AI를 사용했다.

처음부터 과제를 전부 맡기는 방식으로 사용한 것이 아니라, ERDCloud에서 직접 테이블과 관계를 설계한 뒤 좋아요 관계, 이미지 테이블 분리, DDL 제약조건 설정처럼 판단이 필요한 부분을 질문하면서 보완하는 방식으로 활용했다.

---

## 질문한 내용

### 1. ERD 관계 설정 확인

회원, 게시글, 댓글, 이미지, 좋아요 테이블을 직접 설계한 뒤 관계선이 맞는지 확인했다.

특히 좋아요 관계에서 회원과 게시글이 N:M 관계라고 생각했는데, 관계형 데이터베이스에서는 이를 직접 연결하지 않고 `post_likes` 중간 테이블로 분리해야 한다는 점을 질문했다.

AI 답변을 참고해 다음과 같이 정리했다.

* `users` 1 : N `post_likes`
* `posts` 1 : N `post_likes`
* 결과적으로 `post_likes`를 통해 `users`와 `posts`의 N:M 관계를 구현

이를 통해 단순히 N:M이라고 생각하는 것과 실제 테이블로 구현하는 방식이 다르다는 것을 이해했다.

---

### 2. 이미지 테이블 분리 여부 검토

처음에는 게시글에 이미지 정보를 바로 넣는 방식도 생각했다. 하지만 게시글 하나에 이미지가 여러 개 들어갈 수 있고, 이미지도 생성일, 수정일, 삭제일 같은 정보를 따로 가질 수 있다고 판단했다.

AI에게 이미지 테이블을 따로 분리하는 것이 맞는지 질문했고, 게시글과 이미지를 1:N 관계로 분리하는 것이 더 자연스럽다는 답변을 참고했다.

최종적으로 `image` 테이블을 따로 만들고 `post_id`를 외래키로 두었다.

---

### 3. 조회수 컬럼 관계 여부 확인

`posts` 테이블에 `view_count`가 있기 때문에 조회수도 별도의 관계가 필요한지 헷갈렸다.

AI에게 질문한 결과, 조회수는 다른 엔티티와 관계를 맺는 데이터가 아니라 게시글 자체의 속성값이라는 것을 확인했다. 그래서 별도의 관계선을 만들지 않고 `posts` 테이블의 컬럼으로 유지했다.

다만 누가 조회했는지, 언제 조회했는지 같은 조회 이력을 저장하려면 별도의 조회 테이블이 필요하다는 점도 함께 이해했다.

---

### 4. DDL 작성 방식 확인

ERDCloud에서 DDL을 Export했지만, 자료형과 기본값, 제약조건이 부족하게 나와서 그대로 제출하기 어렵다고 판단했다.

AI에게 Export된 DDL을 보여주고 어떤 부분을 보완해야 하는지 질문했다. 이를 통해 다음 내용을 직접 반영했다.

* 기본키에 `AUTO_INCREMENT` 적용
* `created_at`에 `DEFAULT CURRENT_TIMESTAMP` 적용
* `updated_at`에 `ON UPDATE CURRENT_TIMESTAMP` 적용
* `view_count`에 `DEFAULT 0` 적용
* `email`, `nickname`에 `UNIQUE` 적용
* `post_likes(user_id, post_id)`에 복합 `UNIQUE` 적용
* 각 테이블의 외래키 제약조건 확인

특히 `post_likes`에서 `user_id`와 `post_id` 각각을 따로 `UNIQUE`로 설정하는 것이 아니라, 두 컬럼의 조합을 기준으로 중복을 막아야 한다는 점을 확인했다.

---

### 5. DATETIME과 TIMESTAMP 선택 이유 정리

시간 컬럼에 `DATETIME`을 사용할지 `TIMESTAMP`를 사용할지 고민했다.

AI 답변을 참고해 이번 과제는 로컬 환경에서 커뮤니티 테이블 구조를 설계하는 것이 목적이고, UTC나 시간대 변환까지 고려하는 상황은 아니라고 판단했다. 그래서 생성일, 수정일, 삭제일 값을 그대로 저장하는 `DATETIME`을 사용했다.

---

### 6. 회고와 README 작성 보조

ERD 설계 과정에서 어떤 부분을 고민했는지 정리하기 위해 AI에게 문장 구성을 도움받았다.

다만 내용 자체는 내가 실제로 겪은 고민을 기준으로 작성했다.

* 좋아요 관계를 N:M으로만 생각하다가 중간 테이블로 분리한 점
* 이미지 정보를 게시글 컬럼으로 둘지 별도 테이블로 나눌지 고민한 점
* ERDCloud에서 FK 자동 생성 방식을 헷갈렸던 점
* Export된 DDL을 그대로 쓰지 않고 직접 제약조건을 보완한 점
* 설계를 한 번에 끝내지 못하고 여러 번 수정하면서 관계를 검토한 점
