# KTB4_gourmet_Week5

## 과제 개요
커뮤니티 서비스의 ERD를 설계하고, ERD와 동일한 결과를 만들 수 있는 DDL을 작성했다.

## 설계 테이블
- users
- posts
- comments
- image
- post_likes

## ERD 설계 핵심
이번 과제에서는 지난 4주차에 설계했던 커뮤니티 API 구조를 기준으로 실제 데이터베이스 테이블을 설계했다. 단순히 ERD를 그리는 것보다는 각 데이터가 어떤 테이블에 들어가야 하고, 테이블끼리 어떤 관계를 가져야 하는지 고민하는 데 집중했다.

커뮤니티 기능에 필요한 핵심 데이터는 회원, 게시글, 댓글, 이미지, 좋아요라고 판단해서 총 5개의 테이블로 나누었다.

- users : 회원 정보 저장
- posts : 게시글 정보 저장
- comments : 댓글 정보 저장
- image : 게시글 이미지 정보 저장
- post_likes : 게시글 좋아요 정보 저장

설계를 하면서 헷갈렸던 부분은 좋아요 관계였다. 처음에는 회원과 게시글이 서로 여러 개의 관계를 가질 수 있으니까 단순히 N:M 관계라고만 생각했는데, 실제 관계형 데이터베이스에서는 N:M 관계를 바로 표현하지 않고 중간 테이블로 분리해야 한다는 것을 다시 정리했다.

그래서 좋아요는 post_likes 테이블을 따로 만들고, 회원과 좋아요는 1:N, 게시글과 좋아요도 1:N 관계로 설계했다. 결과적으로 post_likes를 통해 회원과 게시글의 N:M 관계를 구현한 구조가 되었다.

게시글과 댓글, 이미지도 각각 독립된 데이터라고 판단했다. 게시글 하나에는 댓글이 여러 개 달릴 수 있고, 이미지도 여러 개 들어갈 수 있기 때문에 comments와 image테이블은 posts의 기본키를 외래키로 참조하도록 설계했다. 댓글은 어떤 회원이 작성했는지도 필요하기 때문에 users도 함께 참조하도록 했다.

DDL을 작성할 때는 ERDCloud에서 Export한 SQL을 그대로 사용하지 않고, 실제 실행 가능한 DDL이 되도록 자료형과 제약조건을 직접 보완했다. 기본키에는 AUTO_INCREMENT를 적용했고, 생성일은 DEFAULT CURRENT_TIMESTAMP를 사용해서 데이터가 생성될 때 자동으로 시간이 들어가도록 했다.

회원의 이메일은 로그인 식별자로 사용될 수 있기 때문에 중복을 허용하지 않도록 UNIQUE제약조건을 추가했다. 닉네임도 서비스 내에서 중복되지 않는 값이라고 가정해서 UNIQUE로 설정했다.

좋아요 테이블에서는 같은 회원이 같은 게시글에 좋아요를 여러 번 누르면 안 되기 때문에 user_id와 post_id를 묶어서 복합 UNIQUE 제약조건을 추가했다. 각각의 컬럼을 따로 UNIQUE로 설정하면 회원이 좋아요를 한 번밖에 못 누르거나, 게시글이 좋아요를 하나밖에 못 받는 잘못된 구조가 되기 때문에 두 컬럼의 조합을 기준으로 중복을 막도록 했다.

게시글의 조회수는 다른 테이블과 관계를 맺는 데이터가 아니라 게시글 자체의 속성이라고 판단했다. 그래서 별도의 관계 테이블을 만들지 않고 posts 테이블 안에 view_count 컬럼을 두었고, 기본값은 0으로 설정했다.

이번 설계를 통해 단순히 테이블을 나누는 것보다, 데이터가 어떤 기준으로 독립되어야 하는지와 관계를 어떻게 풀어내야 하는지가 중요하다는 것을 알게 되었다. 특히 N:M 관계를 중간 테이블로 분리하는 과정과, DDL에서 제약조건을 통해 데이터 무결성을 보장하는 부분을 중점적으로 학습했다.

## 관계 설명
- users 1 : N posts
- users 1 : N comments
- users 1 : N post_likes
- posts 1 : N comments
- posts 1 : N image
- posts 1 : N post_likes

## DDL 설계 핵심
- PK에는 AUTO_INCREMENT 적용
- created_at은 DEFAULT CURRENT_TIMESTAMP 적용
- email, nickname은 UNIQUE 적용
- post_likes는 user_id, post_id 복합 UNIQUE 적용
- view_count는 DEFAULT 0 적용
